# Implicit Feedback Loop — Design Spec

## Problem

Routes are generated by AI (Gemini Flash) + Google Routes API but there is no feedback mechanism. The app generates fresh routes every time without knowing which routes were good or bad. Each generation starts from zero even if the runner has completed that loop a dozen times.

## Approach

Three loosely coupled components that together form a self-improving route system:

1. **Implicit Run Analysis** — compute route adherence and deviation zones after every run
2. **Verified Route Library** — promote well-run routes so they can be reused without AI
3. **AI Prompt Feedback Context** — inject historical deviation data into the AI prompt when generating new routes

All analysis runs client-side. Data lives in IndexedDB. The design is intentionally offline-first and backend-agnostic — Supabase sync is a planned Phase 2.

---

## Component 1: Implicit Run Analysis

After every completed run, the app computes a `RunAnalysis` object and persists it. No user action required.

### What is computed

**Route adherence (0–100%)** — For each GPS trace point, compute the distance to the nearest point on the planned polyline (`CompletedRun.routePolyline`). Adherence = percentage of trace points within 50 m of the route. If `routePolyline` is absent (free run, no planned route), adherence is null and no analysis is created.

**Deviation zones** — Contiguous stretches where the runner was more than 50 m from the planned route for more than 100 m of distance. Each zone is recorded with start/end coordinates and the maximum deviation observed. These are the raw material for future prompt injection.

**Completion** — `trace distance / route distance` as a 0–1 fraction. Below 0.8 is treated as an aborted run and the route is not promoted to verified (see Component 2).

### Data model

```typescript
interface DeviationZone {
  startCoord: [number, number]; // [lng, lat]
  endCoord: [number, number];   // [lng, lat]
  maxDeviation: number;          // meters
  length: number;                // meters along route
}

interface RunAnalysis {
  id: string;
  runId: string;                 // links to CompletedRun.id
  routeId: string | null;        // links to SavedRoute.id — null for unrouted runs
  adherence: number;             // 0–100 percentage
  deviationZones: DeviationZone[];
  completion: number;            // 0–1 fraction
  computedAt: string;            // ISO timestamp
}
```

### Storage

New IndexedDB store: `run_analysis`

- `keyPath: 'id'`
- Index on `runId` (unique)
- Index on `routeId`

### Existing types modified

`CompletedRun` in `src/types/index.ts` — add optional field:

```typescript
analysisId?: string;
```

### Where it runs

Computation happens client-side immediately after the run ends, inside `useRunSession` (or equivalent hook). The flow is:

1. Run ends → `CompletedRun` is persisted
2. `computeRunAnalysis(run)` runs synchronously (or in a microtask)
3. `RunAnalysis` is saved to `run_analysis` store
4. `CompletedRun.analysisId` is updated
5. If `adherence >= 90` and `completion >= 0.9` and `routeId` is set → trigger Component 2 verification check

### New file

`src/lib/run-analysis.ts` — exports:

- `computeRunAnalysis(run: CompletedRun): RunAnalysis | null`
- `saveRunAnalysis(analysis: RunAnalysis): Promise<void>`
- `getRunAnalysis(runId: string): Promise<RunAnalysis | null>`
- `getAnalysesForRoute(routeId: string): Promise<RunAnalysis[]>`
- `getAnalysesNear(lat: number, lng: number, radiusMeters: number): Promise<RunAnalysis[]>`

## Component 2: Verified Route Library

A route becomes "verified" when at least one run meets the quality bar. Verified routes can be served directly instead of triggering AI generation.

### Verification criteria

A single run with `adherence >= 90%` AND `completion >= 0.9` is sufficient to verify a route. Subsequent good runs increase confidence (reflected in `avgAdherence` and `timesRun`) but are not required.

### SavedRoute extended fields

Added to the existing `SavedRoute` interface in `src/lib/storage.ts`:

```typescript
verified?: boolean;
timesRun?: number;
avgAdherence?: number;    // rolling average across all analysed runs
lastRunAt?: string;       // ISO timestamp
```

These fields are updated each time a `RunAnalysis` is linked to the route.

### Matching logic

Before calling AI, the route generation flow checks:

1. Is there a `verified` route whose start point is within 200 m of the requested start?
2. Is its distance within ±15% of the requested distance?

If both conditions match, return that route directly — skip AI and Google Routes API entirely.

If multiple verified routes match, prefer the one with the highest `avgAdherence`, then the most recently run.

Matching does nothing until at least one verified route exists. There is no minimum threshold or warm-up period — the first good run creates the first verified route.

### New file

`src/lib/route-library.ts` — exports:

- `checkRouteLibrary(startLat: number, startLng: number, distanceKm: number): Promise<SavedRoute | null>`
- `updateRouteStats(routeId: string, analysis: RunAnalysis): Promise<void>` — updates `timesRun`, `avgAdherence`, `lastRunAt`, and sets `verified` if criteria met
- `getVerifiedRoutes(): Promise<SavedRoute[]>`

## Component 3: AI Prompt Feedback Context

When generating a new route (no library match), the AI prompt includes a short historical feedback block built from stored `RunAnalysis` data near the requested start point.

### What is injected

**Deviation warnings** — Systematic deviations are zones that appear in multiple runs within 100 m of each other. If any exist within 1 km of the start point, the prompt includes: `Avoid area near [lat], [lng] (runners consistently deviate here)`.

**Preferred segments** — Segments from nearby verified routes with high adherence (>=95%). Expressed as a coordinate pair with a short label: `Preferred path: [lat]→[lat] along waterfront (high adherence)`.

Cap at 3–5 feedback points total to avoid overloading the model.

### Prompt format

```
HISTORICAL FEEDBACK (from real runs near this start):
- Avoid area near 59.3350, 18.0420 (runners consistently deviate here)
- Preferred path: 59.3280→59.3270 along waterfront (high adherence)
```

The block is appended to the existing system prompt before the generation request. If no relevant data exists the block is omitted entirely — no placeholder text.

### New file

`src/lib/prompt-feedback.ts` — exports:

- `buildPromptFeedback(startLat: number, startLng: number): Promise<string>` — returns the formatted feedback block or empty string

## Architecture

```
run ends
  └─ computeRunAnalysis()         [src/lib/run-analysis.ts]
       └─ saveRunAnalysis()
       └─ updateRouteStats()      [src/lib/route-library.ts]
            └─ sets verified if criteria met

route generation requested
  └─ checkRouteLibrary()          [src/lib/route-library.ts]
       ├─ match found → return SavedRoute (skip AI)
       └─ no match → buildPromptFeedback()  [src/lib/prompt-feedback.ts]
                         └─ AI generation with feedback context
```

All three paths are client-side. IndexedDB is the only persistence layer at this stage.

---

## Integration Points

### After run ends (in `useRunSession` or equivalent)

```
CompletedRun persisted
  → computeRunAnalysis(run)
  → saveRunAnalysis(analysis)
  → if (analysis.routeId) updateRouteStats(analysis.routeId, analysis)
  → update CompletedRun.analysisId
```

### Before route generation (in route generation flow)

```
User requests route
  → checkRouteLibrary(startLat, startLng, distanceKm)
  → if match: return immediately
  → else: feedback = await buildPromptFeedback(startLat, startLng)
           prompt = basePrompt + feedback
           → call AI + Google Routes API as usual
```

## Out of Scope

- Backend/cloud storage (Phase 2 with auth, Supabase planned)
- User-facing analytics dashboard (separate spec: "user insights")
- Pace-per-segment analysis
- Manual feedback (ratings, notes) — implicit signals only
- Cross-device sync
