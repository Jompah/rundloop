Interfaces & Datamodeller

Sektion 2 av 4 — Vad varje interface gör

MapRenderer
init(container, options) → Map
setCenter(lat, lng, zoom)
addPolyline(coords, style)
addMarker(lat, lng, options)
fitBounds(bounds)
setHeading(degrees)
destroy()
Notera: Gradient-polyline (elevation coloring) hanteras olika per provider. Interfacet accepterar en colorExpression — varje adapter översätter till sitt format.
RoutingEngine
getRoute(waypoints) → Route
getRouteWithAlternatives(wp) → Route[]
Route-objekt:
{ polyline, distance, duration, instructions[] }

Instructions normaliseras till gemensamt format: { type: "turn-left" | "turn-right" | "continue" | ..., text, distance, location }
Geocoder
reverseGeocode(lat, lng) → Place
search(query, near?) → Place[]
Place-objekt:
{ name, city, country, lat, lng }

Idag används bara reverseGeocode för att hämta stadsnamn. search förberett för framtida "sök destination".
POIProvider
getNaturePOIs(center, radius) → POI[]
getLandmarks(center, radius) → POI[]
getIslandOutline(lat, lng) → Polygon?
POI-objekt:
{ name, type, lat, lng, emoji? }

Obs: Google Places har rikare data (ratings, foton) men vi normaliserar ner till gemensamt format. Extra-data kan exponeras via metadata-fält.

Designbeslut

  • Normaliserade returtyper — alla providers returnerar samma Route, Place, POI-objekt
  • Provider-specifik metadata — extra data (Google ratings, Mapbox confidence scores) i ett valfritt metadata-fält
  • Async allt — alla metoder returnerar Promise (även om Open-implementationen ibland är synkron)
  • Elevation separat — hålls utanför provider-abstraktionen, Open-Meteo fungerar med alla providers
