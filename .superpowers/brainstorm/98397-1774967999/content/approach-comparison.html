Arkitektur-approach: Map Provider Abstraction

Tre alternativ för hur vi bygger adaptermönstret

A

Interface-per-tjänst

Separata interfaces för varje kapabilitet: MapRenderer, RoutingEngine, Geocoder, POIProvider. Varje provider implementerar de interfaces den stödjer. Mix-and-match möjligt (t.ex. Google Maps rendering + OSRM routing).

Fördelar

  • Maximal flexibilitet — testa enskilda delar
  • Kan hitta bästa combo (Google tiles + OSRM routing)
  • Enkelt att lägga till en ny provider för bara en tjänst

Nackdelar

  • Fler interfaces att underhålla
  • Konfiguration mer komplex (4 val istället för 1)
B

Monolitisk provider

Ett enda MapProvider-interface som varje provider implementerar helt: rendering, routing, geocoding, POI. Du byter allt på en gång — "Google-läge" eller "Open-läge".

Fördelar

  • Enkel konfiguration — ett val
  • Garanterat konsistent inom en provider
  • Mindre kod att skriva

Nackdelar

  • Ingen mix-and-match
  • Om en provider saknar en kapabilitet måste du ändå implementera den
  • Svårare att isolera vad som gör skillnad i A/B-test
C

Hybrid: Provider-bundles med override

Varje provider levereras som ett bundle (rendering + routing + geocoding + POI), men enskilda tjänster kan overridas. Default: välj "google" och du får hela Google-stacken. Override: "google" men med OSRM för routing.

Fördelar

  • Enkelt default-val (ett klick) men flexibelt vid behov
  • A/B-test kan jämföra hela bundles ELLER enskilda tjänster
  • Pragmatisk balans mellan A och B

Nackdelar

  • Något mer komplext än B, men inte lika mycket som A
  • Override-logik behöver vara tydlig

💡 Min rekommendation: C — Hybrid

Du får snabb A/B-testning mellan hela stacks (Open vs Google vs Mapbox) men kan också isolera enskilda tjänster om du märker att t.ex. Google-routing är bättre men MapLibre-rendering ser snyggare ut. Det är den approach som ger mest insikt med minst konfigurationsbörda.
